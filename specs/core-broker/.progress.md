## Design Learnings

- Design insight: Fathom `then.assert` action is load-bearing for Nautilus — routing rules write into `routing_decision`/`scope_constraint`/`denial_record` templates rather than emitting `__fathom_decision`. Recommend a step-5 spike to validate the `then.assert` compiler surface before building adapters (recorded as risk in design §17).
- Design insight: Operator allowlist enforced in two places (`Literal[...]` on `ScopeConstraint.operator` + runtime adapter check). Both must be updated together or pyright strict mode will fail — mechanical coupling prevents drift.
- Design insight: `asyncio.run()` per `Broker.request()` has acceptable overhead for Phase 1 (<5 ms per call against Fathom's ~15 µs evaluation budget) but is unacceptable for Phase 2 REST/MCP servers — they will consume `Broker.arequest()` directly.
- Design insight: CLIPS slot typing is restrictive (`string | symbol | float | integer`); multi-valued slots (data_types, allowed_purposes) are encoded as space-separated strings and exploded inside rules via `explode$`. This is why `overlaps()` is a Nautilus-shipped external function rather than a built-in match.
- Design insight: Ephemeral Ed25519 keypair (auto-generated on first run) is acceptable for Phase 1 because attestation verifiers are internal; persistent keystore is Phase 2 work. Documented for operators via `attestation.private_key_path` config.
- Design insight: Field identifier validation (regex `^[A-Za-z_][A-Za-z0-9_]*(\.[A-Za-z_][A-Za-z0-9_]*)?$`) is a second SQL-injection defense layer beyond parameterization — scope constraints can name columns dynamically, so the column name itself must be validated even though values are parameterized.
- Design insight: `EmbeddingUnavailableError` is a distinct failure class from `AdapterError` because it represents a configuration gap (no embedder registered) rather than a runtime fault. Both surface under `sources_errored` but the `error_type` discriminator preserves the distinction for operators.
- Design insight: `BrokerResponse` normalizes four disjoint source outcome sets — `sources_queried` (succeeded), `sources_denied` (policy), `sources_skipped` (not selected by routing), `sources_errored` (runtime fault). Keeping these distinct prevents downstream consumers from conflating operational failures with policy outcomes in dashboards and audit.
