# Brief: operator-platform

## Theme

Operator-facing surfaces and ecosystem completion. Combines design.md "Phase 4 — Ecosystem" with the Admin UI that was lifted out of Phase 3. Every item in this spec is aimed at someone *running* Nautilus rather than the Broker's request path.

## Context

After `reasoning-engine` ships, Nautilus's Broker is feature-complete: full reasoning, all mainstream adapters, REST + MCP transport, attestation dispatch. What's left is everything that makes Nautilus *operable* at scale: observability, policy packs for common compliance regimes, a UI for humans to inspect routing decisions and audit history, ecosystem adapters for niche data sources, and an SDK so third parties can ship their own adapters.

## Dependency on `reasoning-engine`

Hard dependencies:
- Admin UI displays classification-escalation events, cross-agent flow denials, and cumulative-exposure state — all introduced in `reasoning-engine`.
- Rule packs (NIST, HIPAA) rely on classification hierarchy + cumulative exposure primitives.
- Benchmarking suite measures the full broker pipeline including LLM analyzer + all adapters from `reasoning-engine`.
- Custom adapter SDK documents the `Adapter` protocol surface after it has stabilized through 6 reference implementations (core-broker: 2, reasoning-engine: 4).

## In scope

**Ecosystem adapters:**
- InfluxDB adapter with measurement/tag scoping + time range enforcement
- S3/document store adapter with prefix/tag-based access + classification label filtering

**Policy distribution:**
- Rule pack: `data-routing-nist` (NIST SP 800-53 / CUI controls)
- Rule pack: `data-routing-hipaa` (PHI classification + minimum-necessary rules)
- Custom adapter SDK — documented `Adapter` protocol, example adapter repo, author guide

**Observability:**
- Grafana dashboard templates (request rate, routing decision distribution, scope-denial rate, attestation success rate, per-source latency)
- Benchmarking suite — load harness + latency/throughput report across all adapters

**Admin UI:**
- Source status view (per-source health, last-successful-query, scope policy summary)
- Routing decision viewer (per-request trace: facts asserted, rules fired, sources selected/denied/skipped, scope constraints)
- Audit log viewer (filterable by agent, source, decision, time range; attestation token verification UI)
- Authentication (operator-role only; assume upstream SSO / reverse proxy)

## Out of scope

- Any additional reasoning capabilities (those belong in `reasoning-engine` or a future spec)
- LLM provider integrations beyond what `reasoning-engine` delivers
- Multi-tenant features
- Agent-side tooling (agents interact via Broker surface only)

## Success criteria

- All rule packs installable via `fathom.packs` entry-point mechanism; each passes Fathom's pack validation
- Admin UI renders decisions + audit entries generated by the existing `core-broker` + `reasoning-engine` pipelines with zero Broker-side changes (read-only operator tooling)
- Custom adapter SDK includes a working example adapter in a separate repo/package, green CI, and a "write your first adapter" tutorial
- Benchmarking suite produces a reproducible report published with a Docker image tag
- Grafana dashboards render against the existing audit sink output (possibly via a Grafana-compatible exporter if the `audit.jsonl` format needs a bridge)

## Rough task estimate

~30-35 tasks. Smaller than `reasoning-engine` because Phase 4 items are predominantly documentation + config + UI rendering, with no new architectural surfaces in the Broker itself.

## Open questions for research phase

- Admin UI tech choice: React + Vite, HTMX + Jinja, or other? (Driver: air-gap deploy constraints.)
- Authentication model for the UI: API-key-based, OIDC, or defer to reverse proxy?
- Grafana bridge: direct `audit.jsonl` scraping, Prometheus exporter, or OpenTelemetry span emission?
- Custom adapter SDK packaging: same repo as Nautilus, separate `nautilus-adapter-sdk` package, or cookiecutter template only?
- Rule pack distribution: PyPI as standalone packages (`nautilus-rules-nist`) or YAML files under a common `rule-packs/` directory?
