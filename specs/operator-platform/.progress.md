## Learnings
- uv run required to use venv-installed packages (not bare python)
- Adapter Protocol surface is stable: 6 implementations (postgres, pgvector, elasticsearch, neo4j, rest, servicenow), 3 async methods + ClassVar[str]. Safe to extract.
- SDK's only required runtime dep is `pydantic>=2.0`. All types (IntentAnalysis, ScopeConstraint, AdapterResult, SourceConfig, AuthConfig, EndpointSpec, ErrorRecord) are pure Pydantic models.
- Fathom already uses `importlib.metadata` entry points (`fathom.packs` group) for rule pack discovery via `RulePackLoader.discover()`. Nautilus adapter discovery should mirror this pattern under `nautilus.adapters` group.
- SourceConfig.type is a Literal union of built-in adapter names. SDK should relax this to `str` so third-party adapters can declare arbitrary type names.
- Nautilus requires Python >=3.14, but SDK should target >=3.11 for broader reach. All typing features used (Protocol, ClassVar, Literal, runtime_checkable) are available in 3.11+.
- Copier preferred over Cookiecutter for adapter scaffold template due to `copier update` for template evolution sync.
- SQLAlchemy's dialect compliance suite pattern is directly applicable: SDK should ship an importable `AdapterComplianceSuite` that adapter authors can `from nautilus_adapter_sdk.testing import`.
- Do NOT make Nautilus depend on `nautilus-adapter-sdk` (avoids circular dep + air-gap complications). Instead use CI drift-guard test asserting structural equivalence between SDK types and Nautilus-internal types.
- Fathom's built-in compliance packs (nist-800-53, hipaa, cmmc, owasp-agentic) use their OWN templates (access_request, phi_policy, cui_policy, etc.) — completely incompatible with Nautilus's routing templates (agent, source, session, routing_decision). Nautilus data-routing packs must define fresh rules against Nautilus templates.
- Pack validation requires three subdirectories: templates/, rules/, modules/. Optional: functions/ (for hierarchy_ref resolution).
- Entry point registration in pyproject.toml under [fathom.packs] group — e.g., `data-routing-nist = nautilus.rule_packs.data_routing_nist`.
- Salience bands already occupied: 250 (global denials), 200 (purpose-mismatch), 150 (classification), 100 (routing). Compliance rules should use 170-190 (denials), 130-150 (scope constraints), 110-120 (escalations).
- HIPAA minimum necessary standard maps directly to Nautilus scope_constraint template — emit field-level redact/restrict constraints per purpose.
- PHI classification requires tagging sources by which of the 18 HIPAA identifiers they contain (via data_types slot), then rules match on those tags.
- CUI sub-category hierarchy (cui-sub.yaml) only covers 3 categories; NIST pack may need to extend with additional CUI Specified types (ITAR, EAR, FTI, LES).
- Both packs must carry compliance disclaimer (reference implementation only, not certified).
- HTMX 2.0.8 is current stable; 4.0 (fetch-based) expected mid-2026. Vendor 2.0.x for stability.
- HTMX is ~14KB gzip, zero dependencies, single .min.js file — ideal for air-gap vendoring.
- FastAPI's `Jinja2Templates` resolves all paths relative to one directory; use `partials/` and `pages/` subdirs for HTMX fragment vs full-page separation.
- The `HX-Request` header distinguishes HTMX partial requests from full-page loads — same endpoint can serve both.
- `hx-push-url="true"` pushes filter/sort/page state to browser URL — enables bookmarking and back-button for table views.
- SSE via `sse-starlette` + HTMX `hx-ext="sse"` is the cleanest approach for live source status. Audit log viewer should use manual refresh or polling (operators filter/search, not watch live).
- `proxy_trust_dependency` already exists in `nautilus/transport/auth.py` — admin UI auth is zero additional work.
- `decode_nautilus_entry()` in audit/logger.py round-trips JSONL lines back to `AuditEntry` Pydantic models — use this for the audit viewer backend.
- Audit JSONL lines are nested: outer Fathom `AuditRecord` wraps inner Nautilus `AuditEntry` in `metadata.nautilus_audit_entry` — must double-parse (JSON parse outer, extract metadata string, parse inner).
- Admin UI should mount as FastAPI `APIRouter` under `/admin/` on the existing app, not a separate ASGI app — shares broker lifespan and auth.
- Skip Hyperscript (still experimental); vanilla JS onclick handlers suffice for modal close.
- No CSS framework CDN needed; a single custom `styles.css` (or a vendored classless CSS like Pico) keeps air-gap clean.
- `python-multipart` is needed for form data parsing if using POST-based filters. Already a FastAPI optional dep.
- Large audit.jsonl files need seek-based pagination or line-index caching; naive readlines() won't scale to GBs.
- `opentelemetry-instrumentation-fastapi` v0.62b0 (April 2026) supports Python 3.9-3.14 — compatible with Nautilus's >=3.14 requirement.
- OTel FastAPI auto-instrumentation provides request/response hooks (`server_request_hook`, `client_response_hook`) for injecting custom business attributes on auto-generated spans.
- Excluded URLs via `OTEL_PYTHON_FASTAPI_EXCLUDED_URLS` env var (regex CSV) — use for `/healthz,/readyz` probes.
- OTel Python SDK overhead: ~1-3% latency. `BatchSpanProcessor` is async with configurable 5s flush interval and 2048-span queue. Negligible impact on Nautilus's 200ms p95 budget.
- `OTEL_SDK_DISABLED=true` env var makes the entire SDK no-op — ideal for air-gapped mode without code changes.
- `grafanalib` (Python dashboard library) last supports Python <=3.11, effectively unmaintained. Raw JSON dashboard provisioning is the only viable Python-3.14-compatible option for Grafana dashboards-as-code.
- Prometheus metric naming convention: `nautilus_requests_total` (underscore-separated, `_total` suffix for counters, `_bucket` suffix for histograms). OTel SDK auto-converts dot-separated names to underscore format.
- OTel dependencies should be optional (`nautilus[otel]`) to keep base install lean for air-gap deployments. Guard all imports with try/except ImportError.
- design-v2.md already specifies 7 v1 metrics (`nautilus_requests_total`, `nautilus_clips_evaluation_ms`, etc.) — implementation must align with these names.
- Locust integrates with OTel via `--otel` flag for observable load testing. Preferred over custom harness for realistic HTTP benchmarks.
- Grafana datasource provisioning supports Tempo→Prometheus exemplar links and Loki→Tempo derived field drill-down — enables full trace-metric-log correlation out of the box.
- Log correlation requires injecting `trace_id` and `span_id` into structured log records. Consider also injecting into audit.jsonl for Loki→Tempo drill-down from audit entries.
- Requirements: Audit JSONL seek-based pagination is the critical path for Admin UI scalability — naive readlines() won't work for GB-sized files. Byte-offset cursors needed, not line-number-based.
- Requirements: Attestation verify POST is the only POST endpoint in Admin UI — it's read-only (verifies, doesn't write) so it doesn't violate the read-only constraint.
- Requirements: SourceConfig.connection must NEVER be rendered in Admin UI (security: would expose credentials). Only id, type, classification shown.
- Requirements: Both rule packs define fresh rules against Nautilus templates — Fathom's built-in packs are incompatible. This is a hard constraint confirmed during research.
- Requirements: SDK drift-guard test is essential because Nautilus must NOT depend on nautilus-adapter-sdk (avoids circular dep + air-gap issues). Types are duplicated, equivalence enforced by CI.
- Requirements: OTel trace_id injection into AuditEntry is an open question — it requires a broker-side model change which contradicts the "zero broker changes" goal. May need to be Phase 5 or a backward-compatible optional field.
- Requirements: 18 user stories across 6 workstreams (Admin UI: 5, Observability: 3, Adapter SDK: 4, Compliance: 2, Ecosystem Adapters: 2, Infrastructure: 2) with 30 FRs and 18 NFRs — all traceable to brief scope.
- Requirements: Attestation verify POST is the only mutation-looking endpoint but is semantically read-only (verifies JWT, writes nothing) — explicitly exempted from NFR-7 read-only constraint.
- Requirements: SDK repo layout (monorepo vs separate) deferred to design as UQ-3 — affects drift-guard CI complexity and third-party DX.
- Requirements: InfluxDB adapter targets v2+ (Flux) only per UQ-6 — v1 InfluxQL support would require a separate adapter or dual-mode complexity.
- Design: UQ-4 (trace_id in AuditEntry) resolved as optional field `trace_id: str | None = None` — minimal broker change, backward compatible, enables Loki→Tempo drill-down.
- Design: Admin UI mount strategy is `app.include_router()` in existing `create_app()` factory, NOT a sub-application — shares lifespan, broker instance, and auth without cross-process complexity.
- Design: Audit seek-based pagination uses base64-encoded byte offsets as cursors. Invalid/tampered cursors reset to page 1 — no injection vector since offset is validated as integer before seek.
- Design: OTel integration into broker uses context manager pattern (`broker_span()`) wrapping each pipeline stage. Import guards at module level via `_OTEL_AVAILABLE` flag — broker code unchanged when OTel absent.
- Design: SourceConfig.type Literal union in Nautilus must be extended with "influxdb" and "s3" — this is a broker-side change but strictly additive (new union members, no existing behavior changes).
- Design: `sse-starlette` + `python-multipart` are the only new runtime deps for Admin UI. Jinja2 is already a FastAPI transitive dependency.
- Design: Grafana dashboards must use raw JSON provisioning (not grafanalib) due to Python 3.14 incompatibility. Dashboard JSON files ship in `observability/grafana/dashboards/` at repo root, not inside the Python package.
- Design: Both NIST and HIPAA packs share salience bands (170-190 denial, 130-150 scope, 110-120 escalation) and are designed to coexist without cross-pack rule conflicts.
- Task planning: 71 POC tasks (Phase 1), 8 refactoring tasks (Phase 2), 11 testing tasks (Phase 3), 7 quality gate tasks + 3 VE tasks (Phase 4), 4 PR lifecycle tasks (Phase 5) = ~104 total tasks including quality checkpoints.
- Task planning: Admin UI is the critical path — 25 tasks in Phase 1 alone (1.4–1.25). OTel is second (1.26–1.32). SDK and rule packs are parallelizable after those.
- Task planning: pyproject.toml must be modified in tasks 1.1 and 1.2 — these are sequential because they both modify the same file. All subsequent module tasks are independent.
- Task planning: SDK package lives under `packages/nautilus-adapter-sdk/` as monorepo workspace member. Import verification commands use `cd packages/nautilus-adapter-sdk` prefix since SDK has its own package structure.
- Task planning: Static asset vendoring (htmx.min.js) may need manual download during execution — placeholder files acceptable for POC, real vendored assets required for Phase 4.
- Task planning: VE tasks test admin UI pages via curl against running dev server. Auth may need to be bypassed or a test API key configured in nautilus.yaml for VE2 to succeed.
- Task planning: Rule pack YAML files are validated structurally (YAML parse) in POC phase. Fathom Engine.load_pack() validation deferred to Phase 3 integration tests.
- Task planning: Broker.py receives OTel span instrumentation AND entry-point discovery — two separate concerns in one file. Tasks 1.31 and 1.68 are sequential.
